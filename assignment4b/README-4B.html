Java Fork-Join Framework, Barrier Synchronizers, and Readers/Writer Synchronizers (Assignment 4 - Part B)

Objectives and Overview of the Assignment

The objectives of this assignment are to learn how to synchronize concurrent Being gazes via a readers-writer lock (implemented using a Java StampedLock) and a fair semaphore (implemented using your solution to assignment 3b). Videos describing Java StampedLocks appear here and here. I recommend you watch these videos to ensure you understand Java StampedLocks. Naturally, this topic is covered in class, as well.

The fair semaphore and a Java StampedLock are used to enhance the Java Fork-Join framework implementation of the PalantiriSimulator app from Assignment 4A. Assignment 4B focuses on the PalantiriManager, which is used to restrict the number of Beings from Middle-Earth who can concurrently gaze into a fixed number of Palantiri.

Assignment Structure and Functionality

The PalantiriSimulator app is packaged as a project using the latest version of Android Studio. This app is written in both Kotlin and Java and demonstrates many Android capabilities. For the purposes of Assignment 4B, however, you only need to be aware of the following three directories:

  • simulator/managers/palantiri/stampedLockFairSemaphore, which contains the skeleton(s) you'll need to fill in, as described below.

  • app/src/test, which is a set of unit tests that exercise many PalantiriSimulator features and can be used to help test the functionality you implement for this assignment.

  • app/src/androidTest, which is an Android Studio instrumentation test that runs your app automatically.

You'll also need to integrate your enhanced solution for the forkJoinPool folder from Assignment 4A into the latest/greatest forkJoinPool skeleton and use it as the implementation of the BeingManager for Assignment 4B (please address all my comments from Assignment 4A into your Assignment 4B solution).

To compile this code you need to use the provided Android Studio project. You can run this project by clicking the green "Run 'app'" button in the Android Studio IDE, which should automatically select an Android Emulator to run, assuming you have one created. The app's minimum API is 29, so you will need to select an emulator that uses an API >= 29, though I recommend you use API 30. If you don't already have an emulator created you can create it by clicking on the "AVD Manager" button in the Android Studio IDE.

Program Description and "TODO" Tasks

You'll need to modify several files containing the skeleton Java code by completing the "TODO - You fill in here" tasks to provide a working solution. DO NOT CHANGE THE OVERALL STRUCTURE OF THE SKELETON - just fill in the "TODO" tasks (and do NOT delete the "TODO" tags)!!! In particular, you'll need to finish implementing the following "TODO" tasks for this assignment in the stampedLockFairSemaphore folder:

  • stampedLockFairSemaphoreMgr.java. You'll need to complete the "TODO" tasks in various methods to implement a PalantiriManager via your FairSemaphore implementation, a StampedLock, and a HashMap.

  • FairSemaphoreMO.java or FairSemaphoreCO.java. You'll need to complete the "TODO" tasks in various methods to implement the counting semaphore (implemented using your solution to assignment 3b) with either a built-in Java monitor object (if you're an undergraduate) or a Java Condition Object and ReentrantLock (if you're a graduate student).

  • Student.kt. You need to complete the "TODO" task to set the type field to either Graduate or Undergraduate, depending on which version of the assignment that you're implementing.

Your app will be considered correct if it (1) passes all the unit and instrumentation tests and (2) all Beings successfully complete all their N iterations. A correct simulation should restrict the number of gazing Beings to the number of Palantiri. In other words, if there are four Palantiri in the simulation, then only four Beings should ever be ``gazing'' on the screen at a time.

When the simulation is running, the app view will display visual feedback to show Palantiri and Being states, as well as the progress of their gazing iterations. If your assignment implementation is correct---and does not throw any exceptions---then the app title bar labels will appear green when the simulation completes. If your solution is not implemented correctly---or throws an exception---then red title bar labels will be displayed when the simulation completes.

Concluding Remarks

This part of the assignment is designed to deepen your understanding of synchronizing multi-threaded programs using Java StampedLocks, which are more powerful, more complex, and (sometimes) more efficient readers-writer synchronizers compared with the spin-locks and ReentrantLocks we covered in previous assignments.
